# CS 250 Portfolio

## How do I interpret user needs and implement them into a program? How does creating user stories help with this?

Before I start building anything, I need to know what the user is actually looking for. If I don't, I'm just guessing and that usually leads to extra work later. User stories made that a lot easier because they broke everything into smaller pieces instead of looking at one big project. They also gave me a better idea of what each feature needed to do before I started working on it.

## How do I approach developing programs? What Agile processes do I hope to incorporate into my future development work?

I like taking things one step at a time instead of trying to finish a whole project all at once. Working in sprints made it easier to stay organized and keep making progress. I also liked getting feedback while the project was still being worked on instead of waiting until everything was finished. Going forward, I'd keep using user stories, sprint planning, and retrospectives because they helped keep everything moving and made it easier to fix problems along the way.

## What does it mean to be a good team member in software development?

Being a good teammate is really just doing your part and keeping everyone in the loop. If something doesn't make sense, ask about it. If you find a problem, speak up instead of hoping someone else catches it. Everyone has a different job, but you're all working toward the same goal. I think that's one of the biggest things I got out of this course. Good software comes from people working together, not just from one person writing good code.
